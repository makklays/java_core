
Тестирование в стиле Google (Java)

В Google действует правило: инженер сам пишет тесты для своего кода. Код без тестов не пройдет ревью (Code Review). Вам нужно знать три уровня пирамиды тестирования и специфику их реализации на Java.


1. Unit-тесты (Модульные)

- JUnit 5 / Truth: Изучите стандартный JUnit 5. Также Google использует собственную библиотеку утверждений Google Truth (вместо AssertJ/Hamcrest). Она делает код тестов более читаемым.
Пример: assertThat(response.getDangerScore()).isGreaterThan(90);

- Паттерн Fake вместо Mockito: В Google предпочитают создавать легкие «подделки» (Fakes) тяжелых систем (например, FakeBigtableRepository на базе ConcurrentHashMap), а не засыпать код аннотациями @Mock из Mockito. 
Моки (Mock) используют только для проверки поведения, которое сложно воспроизвести иначе.

- Тестирование многопоточности: Как протестировать, что ваш асинхронный код (на CompletableFuture) корректно отрабатывает конкурентные запросы.

2. Integration-тесты (Интеграционные)

- Testcontainers: Это стандарт индустрии. Позволяет прямо из Java-теста поднять реальный Docker-контейнер с базой данных или эмулятором Spanner/Bigtable, прогнать тесты на реальной БД и потушить контейнер.
- gRPC In-Process Testing: Как протестировать gRPC-сервис без поднятия реальной сети. Библиотека gRPC для Java умеет запускать сервер и клиент внутри одной памяти процесса (InProcessServerBuilder 
  и InProcessChannelBuilder), что делает интеграционные тесты API молниеносными.

3. Тестирование надежности и производительности

- JMH (Java Microbenchmark Harness): Инструмент для замера производительности отдельных участков кода (например, алгоритма парсинга логов). В Google критически важно знать, сколько наносекунд выполняется ваш метод.
- Fuzz-тестирование: Передача в методы случайных, «ломающих» данных для проверки устойчивости системы (особенно важно в Cloud Security).
