Паттерны проектирования (Design Patterns) на Java

На собеседовании в Google вас не попросят «рассказать по книжке про паттерн Абстрактная Фабрика». Вас поставят в условия задачи 
(например, спроектировать гибкую систему правил для анализа вредоносного ПО), где применение правильного паттерна будет единственным логичным решением.


1. Порождающие паттерны (Creational)

- Builder (Строитель): В Google это паттерн №1. Все Protobuf-сообщения в Java создаются только через Builder (MyResponse.newBuilder().build()). Вы должны уметь писать его с закрытыми глазами.
- Singleton (Одиночка): Забудьте про классический синглтон с getInstance(). В Google управление жизненным циклом объектов отдают DI-фреймворку Google Guice. Вам нужно понимать, как работает аннотация @Singleton в контексте контейнера зависимостей.

2. Структурные паттерны (Structural)

- Adapter (Адаптер): Фундамент гексагональной архитектуры. Как обернуть клиент сторонней библиотеки или gRPC-клиент в интерфейс вашей бизнес-логики.
- Decorator (Декоратор): Часто используется для добавления сквозной логики (например, логирования, замера времени выполнения или проверки прав доступа) вокруг gRPC-вызовов (в gRPC это называется ClientInterceptor / ServerInterceptor).

3. Поведенческие паттерны (Behavioral)

- Strategy (Стратегия): Идеально для систем безопасности. Например, у вас есть интерфейс ThreatAnalysisStrategy, и разные реализации: StaticAnalysis, DynamicSandboxAnalysis, AIModelAnalysis. 
  Система выбирает нужную стратегию на лету в зависимости от типа файла.
- Chain of Responsibility (Цепочка обязанностей): Применяется для конвейеров фильтрации данных (Data Pipelines) или цепочек валидации логов безопасности, где запрос последовательно проходит через массив проверок.
- Observer (Наблюдатель): Понимание этого паттерна необходимо для работы со стримами в gRPC (StreamObserver) и реактивного программирования.

4. Дополнительные 3 паттерна, которые нужно знать для Google Cloud

- Circuit Breaker (Предохранитель) — Архитектурный
  Зачем нужен: Защищает микросервисы от каскадных сбоев. Если один сервис (например, банк-партнер в Испании) начинает тормозить или падать, Circuit Breaker временно «размыкает цепь» и мгновенно возвращает дефолтный ответ 
  (или ошибку), не перегружая упавший сервис и не заставляя API Gateway бесконечно ждать таймаута.
  Где смотреть: В Spring вы использовали Resilience4j. В Google нужно понимать саму концепцию — как бы вы написали такой переключатель состояний (CLOSED, OPEN, HALF-OPEN) на чистой Java.
- Read-Write Lock / Thread Pool (Паттерны параллелизма) — Многопоточность
  Зачем нужен: Для раундов по Concurrency. Например, паттерн Thread Pool (который в Java реализован через ExecutorService) — нужно четко понимать, как он управляет очередью задач. 
  Паттерн Read-Write Lock (ReentrantReadWriteLock) — когда много потоков одновременно читают структуру данных и только один пишет.
- Proxy (Заместитель) — Структурный
  Зачем нужен: Это фундамент, на котором работают все перехватчики (Interceptors) в gRPC, логирование, безопасность и транзакции. Вы должны понимать, как работает динамический прокси в Java (java.lang.reflect.Proxy), 
  чтобы объяснить, как gRPC под капотом подменяет сетевой вызов локальным интерфейсом.

Резюме: Ваш финальный список из 10 паттернов

1. Builder (Порождающий) — создание Protobuf-объектов.
2. Singleton/Dependency Injection (Порождающий) — управление жизненным циклом через Guice.
3. Adapter (Структурный) — стыковка слоев в гексагональной архитектуре.
4. Decorator (Структурный) — добавление логирования/валидации к gRPC вызовам.
5. Proxy (Структурный) — перехватчики сетевых запросов.
6. Strategy (Поведенческий) — подмена алгоритмов анализа угроз на лету.
7. Chain of Responsibility (Поведенческий) — конвейеры фильтрации данных и логов.
8. Observer/Reactive Streams (Поведенческий) — работа с gRPC StreamObserver и очередями сообщений.
9. Circuit Breaker (Микросервисный) — отказоустойчивость распределенных систем.
10.Thread Pool / Thread-safe Object (Конкурентность) — высоконагруженная многопоточность.

Как их учить для Google?

Не учите схемы классов. Учите их через проблему. Задавайте себе вопрос: «Какую проблему решает этот паттерн? Что произойдет с моим кодом, если я его НЕ применю (какой принцип SOLID нарушится)?»
