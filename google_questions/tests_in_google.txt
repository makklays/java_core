
В Google действует строгое правило: разработчик сам тестирует свой код. Около 80% всех тестов, создаваемых Java-инженером, составляют малые (Unit) тесты.

1. Архитектура тестов и фреймворки

Фреймворк тестирования: Исключительно JUnit (в основном JUnit 4 со специальными внутренними раннерами от Google, а также JUnit 5).
Библиотека утверждений (Assertions): Вместо стандартного Assert.assertEquals из JUnit или стороннего AssertJ, инженеры Google 
используют собственную библиотеку с открытым исходным кодом — Truth. Она делает логи тестов максимально читаемыми (например, 
assertThat(user.getEmail()).contains("@google.com");).
Изоляция (Тестовые двойники): Для подмены внешних зависимостей (баз данных, удаленных сервисов) используются Mockito и внутренние 
реализации легковесных заглушек (Fakes). Использование «живых» баз данных в малых тестах строго запрещено.

2. Сборка и запуск тестов через Bazel (Blaze)

Google использует монорепозиторий и собственную систему сборки Blaze (её открытая версия называется Bazel). Инженер описывает 
конфигурацию тестов прямо в конфигурационных файлах BUILD:
Для каждого тестового Java-класса объявляется правило java_test.
Команда bazel test //my/project/... автоматически находит и запускает все тесты в компоненте.
Герметичность и кэширование: Тесты в Bazel выполняются в изолированных «песочницах». Если код Java-класса и его зависимости не менялись, 
Bazel вообще не будет перезапускать тест, а мгновенно вернет результат из кэша. Это экономит инженерам часы времени ежедневно.

3. Культура написания кода: Специфические подходы
Работа Java SWE в Google подчиняется жестким инженерным гайдлайнам, которые проверяются на этапе Code Review:
Приоритет публичного API: Тесты пишутся только против публичных методов классов. Тестирование приватных методов через рефлексию 
считается антипаттерном («хрупкие тесты»), так как это мешает проводить рефакторинг.
Тестирование состояния, а не взаимодействия: Разработчиков поощряют проверять конечный результат работы Java-объекта (State-based 
testing), а не то, сколько раз внутри него вызвался метод заглушки через verify() в Mockito (Interaction-based testing).
Мутационные тесты: Специальные внутренние роботы автоматически меняют знаки > на < или + на - в Java-коде инженера. Если после 
этого тесты разработчика не упали, система сообщает, что его тесты «слабые» и не покрывают крайние случаи.

4. Интеграционное тестирование и микросервисы
Поскольку большинство Java-приложений в Google работают в виде облачных микросервисов (внутри инфраструктуры Borg, предка Kubernetes), 
для средних и больших тестов применяются:
gRPC/Protobuf тесты: Вся коммуникация между сервисами идет через gRPC. Инженеры пишут тесты, которые поднимают локальный gRPC-сервер 
в памяти (In-process) для тестирования контрактов API без отправки реальных пакетов по сети.
Hermetic Тесты: Если Java-коду нужно проверить работу с базой данных (например, Cloud Spanner), используются специальные легковесные 
локальные эмуляторы баз данных, запускаемые прямо в контейнере тестирования.
