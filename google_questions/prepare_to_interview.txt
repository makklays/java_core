1. Как будет проходить Coding Round (на Java)

На алгоритмических раундах вам разрешат использовать стандартную библиотеку (JDK). Google оценивает, насколько глубоко вы знаете язык и умеете ли писать эффективный код без использования сторонних фреймворков.
- Коллекции изнутри: Вас точно проверят на знание сложности операций (O(1) vs O(N)). Нужно четко понимать устройство HashMap (коллизии, treeify-порог), ArrayList, PriorityQueue (очень часто нужна для задач на кучу/heap) и Deque.
- Примитивы против оберток: Помните про цену автобоксинга (int vs Integer). В высоконагруженных задачах Google создание лишних объектов в циклах ухудшает производительность и нагружает Garbage Collector.
- Строки: Понимание StringBuilder при манипуляциях со строками в алгоритмах.
- Чистый код: Google требует использовать стандарты кодирования (близкие к Google Java Style Guide). Имена переменных должны быть говорящими, никаких int a, int b, если это не индексы i, j.

2. Специфика System Design (с акцентом на Java/Google)

Поскольку вакансия связана с обработкой гигантских объемов данных о киберугрозах (threat intelligence data), на раунде системного дизайна от вас будут ждать понимания следующих концепций:
- Concurrency & Multithreading (Многопоточность): Это «конек» Java-разработчиков в Google. Спросят про CompletableFuture, ExecutorService, атомики (AtomicInteger), ReentrantLock и ключевое слово volatile. 
Могут попросить спроектировать потокобезопасный кэш или rate-limiter на чистой Java.
- Память и Garbage Collection (GC): Как работает куча (Heap), чем отличаются сборщики мусора (G1, ZGC). Как бы вы оптимизировали сервис на Java, который сталкивается с долгими паузами Stop-The-World (STW) при обработке миллионов логов безопасности?
- Архитектурный паттерн: В Google Cloud микросервисы общаются через gRPC и Protocol Buffers (Protobuf), а не через классический REST/JSON. На дизайне систем обязательно упомяните gRPC как транспорт для высокой производительности и низкого latency.

3. Фронтенд-составляющая для Java-инженера

Так как позиция Full-Stack, на дизайне систем или на отдельном раунде вас спросят, как бэкенд на Java взаимодействует с фронтендом (TypeScript/HTML).
- Стриминг данных: Как передавать терабайты логов угроз на UI в реальном времени? (Ответ: WebSockets или Server-Sent Events вместо постоянного HTTP-поллинга).
- Безопасность API: Архитектура аутентификации (OAuth2, JWT), защита от XSS и CSRF атак — для команды Cloud Security это критично.

вместо привычных Hibernate/JPA. На собеседовании пишите чистый, понятный код на Core Java, фокусируясь на структурах данных и многопоточности.

Как запустить Spanner и Bigtable у себя на Ubuntu?

Вам не нужно платить за облако Google Cloud или подключать банковскую карту. Google выпускает официальные эмуляторы этих баз данных, которые работают локально внутри Docker-контейнеров на Ubuntu.

Запуск Google Cloud Spanner (Локально)
Запустите контейнер с эмулятором:
# docker run -p 9010:9010 -p 9020:9020 gcr.io/cloud-spanner-emulator/emulator
Теперь в коде на Java вы можете подключиться к нему. Для этого в коде достаточно указать, что вы работаете с эмулятором через переменную окружения SPANNER_EMULATOR_HOST=localhost:9010. 
Google Cloud SDK для Java автоматически поймет, что нужно слать запросы на локальный докер, а не в реальное облако.

Запуск Google Cloud Bigtable (Локально)
через Docker:
# docker run -p 8086:8086 google/cloud-sdk:latest gcloud beta emulators bigtable start --host-port=0.0.0.0:8086
В Java-коде укажите переменную окружения BIGTABLE_EMULATOR_HOST=localhost:8086, чтобы перенаправить трафик библиотеки на локальный порт.

Treeify-порог в HashMap — что это?

В Java 8 разработчики оптимизировали HashMap для защиты от DoS-атак (когда злоумышленник специально подбирает ключи с одинаковым hashCode(), вызывая коллизии).

- Как было раньше: При коллизии элементы складывались в связный список (Linked List). Если коллизий много, поиск по кэшу замедлялся с O(1) до O(N).
- Как стало: Если в одной корзине (bucket) собирается слишком много элементов, список автоматически превращается в Красно-черное дерево (Red-Black Tree). В дереве скорость поиска составляет O(log N), что гораздо быстрее при больших объемах данных.

Константы, которые нужно знать:
- TREEIFY_THRESHOLD = 8: Если в одной корзине становится больше 8 элементов, список превращается в дерево (происходит treeify).
- UNTREEIFY_THRESHOLD = 6: Если при удалении элементов из корзины их количество падает до 6, дерево превращается обратно в обычный связный список. Порог равен 6, а не 7, чтобы избежать постоянной «туда-сюда» перестройки, если элементы то добавляются, то удаляются вокруг цифры 8.
- MIN_TREEIFY_CAPACITY = 64: Перестройка в дерево начнется только если общий размер всей HashMap (количество бакетов) не меньше 64. Если бакетов меньше, Java просто увеличит размер самой таблицы (resize).

G1 vs ZGC — в чем разница?

Оба сборщика мусора созданы для работы с большими объемами памяти (память бэкенда в Google), но у них принципиально разная философия и архитектура.

Если говорить простым языком, то ключевое отличие между ними — это длительность пауз в работе программы (Stop-The-World) и то, как они распределяют ресурсы вашего процессора.
Вот детальный разбор разницы, который идеально подойдет для ответа на архитектурном интервью в Google:

Главная разница: Время паузы (Latency)
- G1 (Garbage-First): Этот сборщик пытается балансировать между скоростью работы программы и длительностью пауз. Он останавливает ваше приложение (пауза Stop-The-World), чтобы почистить память старого поколения. На больших объемах данных эти паузы могут длиться от 10 до 200 миллисекунд (а иногда и дольше). Для пользователя это выглядит как микрофриз интерфейса или задержка ответа сервера.
- ZGC (Z Garbage Collector): Это ультрасовременный сборщик (появился как production-ready в Java 15, а в Java 21 получил мощное обновление). Его главная фишка — паузы всегда длятся меньше 1 миллисекунды, причем эта цифра вообще не зависит от того, сколько гигабайт или терабайт памяти выделено вашему приложению.

Как они работают внутри (В чем фокус)
- Как чистит G1: Он делит всю память на равные регионы. Когда приходит время сборки мусора, он выбирает те регионы, где накопилось больше всего «мусора» (отсюда и название Garbage-First), останавливает приложение и начинает переносить живые объекты в другие регионы, освобождая память.
- Как чистит ZGC: Он делает практически всю тяжелую работу (поиск мусора, разметка и даже перемещение живых объектов в памяти) параллельно с работой вашего кода. Программа не останавливается. Достигается это за счет умной технологии Colored Pointers (цветные указатели) и Load Barriers (барьеры чтения): когда ваш код пытается прочитать объект, который ZGC прямо сейчас двигает в памяти, барьер на лету корректирует ссылку.

Обратная сторона медали (Пропускная способность и CPU)
- G1 эффективнее расходует процессор. Поскольку он останавливает приложение на время чистки, процессор занимается только сборкой мусора. Общая пропускная способность (Throughput) у G1 часто выше — приложение в общей сложности успевает сделать больше полезной работы за час, если закрыть глаза на редкие паузы.
- ZGC «жрет» больше CPU. Так как он работает параллельно с вашим кодом, он забирает под свои нужды несколько ядер процессора прямо во время выполнения бизнес-логики. Из-за этого общая производительность приложения может упасть на 2–5% по сравнению с G1, зато у вас никогда не будет задержек.

Объемы памяти
- G1 создавался как замена старому CMS для кучи (Heap) размером от 4 до 64 ГБ.
- ZGC — это монстр для масштабных систем (как раз уровня Google Cloud). Он изначально проектировался под гигантские сервера с объемом памяти до 16 терабайт.

Что выбрать для Google Cloud Security?
Если вы пишете бэкенд, который в реальном времени анализирует критические угрозы безопасности, обрабатывает миллионы сетевых пакетов в секунду и должен мгновенно реагировать на атаки, пауза в 100 мс от G1 может быть фатальной (за это время прилетит еще миллион логов). Поэтому в таких чувствительных к задержкам системах Google делает ставку на ZGC.
